STATIC-001 File System Access in CORS Middleware Medium
File: src/helpers/cors.ts
Line: 21
CORS configuration reads .env file directly from filesystem using require('fs'). This approach bypasses Next.js environment variable handling and could expose secrets if file permissions are misconfigured.
Deep Dive Result
The file reads from 'secrets/.env' at runtime. While the Docker volume mounts this as read-only (:ro), direct filesystem access is less secure than using process.env.
Recommendation
Use process.env for environment variables instead of direct file system access. Configure environment variables through Docker or Next.js mechanisms.
STATIC-002 Excessive Debug Logging Medium
File: src/helpers/cors.ts
Line: 45
Multiple console.log statements expose CORS configuration including allowed origins in production logs. This could leak security-sensitive configuration.
Deep Dive Result
Debug logs include 'ALLOWED_ORIGINS env', 'ENABLE_DEV_CORS env', and specific origin values. These should not be in production.
Recommendation
Remove or conditionally disable debug logging in production environment. Use structured logging with appropriate log levels.
STATIC-003 Permissive CORS for Null Origin Low
File: src/helpers/cors.ts
Line: 38
CORS allows requests with no origin header (null origin). While necessary for Obsidian plugin (app:// protocol), this could be exploited by malicious actors.
Deep Dive Result
Required for Obsidian desktop plugin functionality. The API only exchanges OAuth tokens, limiting abuse potential. Additional validation exists for refresh tokens and auth codes.
Recommendation
Document this behavior and consider additional request validation for requests without origin.
STATIC-004 Sensitive Environment Variable Exposure High
File: src/helpers/debug-env.ts
Line: 24
Debug function logs first 10 characters of CLIENT_SECRET. Even partial secrets can aid attackers in pattern analysis.
Deep Dive Result
Although protected by NODE_ENV === 'development' check, this file should not exist in production builds.
Recommendation
Remove debug-env.ts from production builds or ensure it's completely stripped during build process.
STATIC-005 Redirect URI from Request Header High
File: src/pages/api/tokens.ts
Line: 31
redirect_uri is constructed using req.headers.origin. An attacker could potentially manipulate this header to redirect tokens to malicious endpoints.
Deep Dive Result
Google OAuth validates redirect_uri against pre-registered URIs, providing defense in depth. However, relying on client-supplied origin header is not best practice.
Recommendation
Use server-side configured redirect URI from environment variables instead of request headers.
STATIC-006 Token Stored in Client State Medium
File: src/helpers/ky.ts
Line: 16
Access tokens and refresh tokens are stored in client-side state (hookstate with localStorage persistence). This exposes tokens to XSS attacks.
Deep Dive Result
Tokens are persisted using @hookstate/persistence to localStorage. If XSS occurs, attacker can steal both access and refresh tokens.
Recommendation
Consider using httpOnly cookies for token storage, or implement additional XSS protections. Ensure CSP is properly configured.
STATIC-007 Refresh Token Displayed in UI Low
File: src/pages/index.tsx
Line: 22
The refresh token is displayed to users in an input field for copying. While this is the intended functionality, it increases exposure risk.
Deep Dive Result
This is by design for the Obsidian plugin integration flow. The token is needed for plugin configuration. CSP and security headers provide some protection.
Recommendation
Add clear warnings about token security. Consider time-limited display or copy-once functionality.
